40  * Low-level Deflate/Inflate implementation notes:
41 
42  Compression: Use the "tdefl" API's. The compressor supports raw, static,
43  and dynamic blocks, lazy or
44  greedy parsing, match length filtering, RLE-only, and Huffman-only streams.
45  It performs and compresses
46  approximately as well as zlib.
47 
48  Decompression: Use the "tinfl" API's. The entire decompressor is
49  implemented as a single function
50  coroutine: see tinfl_decompress(). It supports decompression into a 32KB
51  (or larger power of 2) wrapping buffer, or into a memory
52  block large enough to hold the entire file.
53 
54  The low-level tdefl/tinfl API's do not make any use of dynamic memory
55  allocation.
56 
57  * zlib-style API notes:
58 
59  miniz.c implements a fairly large subset of zlib. There's enough
60  functionality present for it to be a drop-in
61  zlib replacement in many apps:
62  The z_stream struct, optional memory allocation callbacks
63  deflateInit/deflateInit2/deflate/deflateReset/deflateEnd/deflateBound
64  inflateInit/inflateInit2/inflate/inflateReset/inflateEnd
65  compress, compress2, compressBound, uncompress
66  CRC-32, Adler-32 - Using modern, minimal code size, CPU cache friendly
67  routines.
68  Supports raw deflate streams or standard zlib streams with adler-32
69  checking.
70 
71  Limitations:
72  The callback API's are not implemented yet. No support for gzip headers
73  or zlib static dictionaries.
74  I've tried to closely emulate zlib's various flavors of stream flushing
75  and return status codes, but
76  there are no guarantees that miniz.c pulls this off perfectly.
77 
78  * PNG writing: See the tdefl_write_image_to_png_file_in_memory() function,
79  originally written by
80  Alex Evans. Supports 1-4 bytes/pixel images.
81 
82  * ZIP archive API notes:
83 
84  The ZIP archive API's where designed with simplicity and efficiency in
85  mind, with just enough abstraction to
86  get the job done with minimal fuss. There are simple API's to retrieve
87  file information, read files from
88  existing archives, create new archives, append new files to existing
89  archives, or clone archive data from
90  one archive to another. It supports archives located in memory or the
91  heap, on disk (using stdio.h),
92  or you can specify custom file read/write callbacks.
93 
94  - Archive reading: Just call this function to read a single file from a
95  disk archive:
96 
97  void *mz_zip_extract_archive_file_to_heap(const char *pZip_filename,
98  const char *pArchive_name,
99  size_t *pSize, mz_uint zip_flags);
100 
101  For more complex cases, use the "mz_zip_reader" functions. Upon opening an
102  archive, the entire central
103  directory is located and read as-is into memory, and subsequent file
104  access only occurs when reading individual files.
105 
106  - Archives file scanning: The simple way is to use this function to scan a
107  loaded archive for a specific file:
108 
109  int mz_zip_reader_locate_file(mz_zip_archive *pZip, const char *pName,
110  const char *pComment, mz_uint flags);
111 
112  The locate operation can optionally check file comments too, which
113  (as one example) can be used to identify
114  multiple versions of the same file in an archive. This function uses a
115  simple linear search through the central
116  directory, so it's not very fast.
117 
118  Alternately, you can iterate through all the files in an archive
119  (using mz_zip_reader_get_num_files()) and
120  retrieve detailed info on each file by calling mz_zip_reader_file_stat().
121 
122  - Archive creation: Use the "mz_zip_writer" functions. The ZIP writer
123  immediately writes compressed file data
124  to disk and builds an exact image of the central directory in memory.
125  The central directory image is written
126  all at once at the end of the archive file when the archive is finalized.
127 
128  The archive writer can optionally align each file's local header and file
129  data to any power of 2 alignment,
130  which can be useful when the archive will be read from optical media.
131  Also, the writer supports placing
132  arbitrary data blobs at the very beginning of ZIP archives. Archives
133  written using either feature are still
134  readable by any ZIP tool.
135 
136  - Archive appending: The simple way to add a single file to an archive is
137  to call this function:
138 
139  mz_bool mz_zip_add_mem_to_archive_file_in_place(const char *pZip_filename,
140  const char *pArchive_name,
141  const void *pBuf, size_t buf_size, const void *pComment,
142  mz_uint16 comment_size, mz_uint level_and_flags);
143 
144  The archive will be created if it doesn't already exist, otherwise it'll
145  be appended to.
146  Note the appending is done in-place and is not an atomic operation, so if
147  something goes wrong
148  during the operation it's possible the archive could be left without a
149  central directory (although the local
150  file headers and file data will be fine, so the archive will be
151  recoverable).
152 
153  For more complex archive modification scenarios:
154  1. The safest way is to use a mz_zip_reader to read the existing archive,
155  cloning only those bits you want to
156  preserve into a new archive using using the
157  mz_zip_writer_add_from_zip_reader() function (which compiles the
158  compressed file data as-is). When you're done, delete the old archive and
159  rename the newly written archive, and
160  you're done. This is safe but requires a bunch of temporary disk space or
161  heap memory.
162 
163  2. Or, you can convert an mz_zip_reader in-place to an mz_zip_writer
164  using mz_zip_writer_init_from_reader(),
165  append new files as needed, then finalize the archive which will write an
166  updated central directory to the
167  original archive. (This is basically what
168  mz_zip_add_mem_to_archive_file_in_place() does.) There's a
169  possibility that the archive's central directory could be lost with this
170  method if anything goes wrong, though.
171 
172  - ZIP archive support limitations:
173  No zip64 or spanning support. Extraction functions can only handle
174  unencrypted, stored or deflated files.
175  Requires streams capable of seeking.
